# IronCrown AI 协作总则

本文件适用于 ChatGPT、Claude、Codex、OpenClaw、DeepSeek 及其他参与 IronCrown 的 AI 或自动化代理。

## 1. 唯一责任人

产品负责人：罗松涛。

只有产品负责人可以最终批准：

- 产品定位变化。
- MVP 范围变化。
- 核心架构变化。
- Unity 大版本更换。
- 存档格式破坏性变更。
- 联机协议破坏性变更。
- 商业化与付费设计。
- 正式发布、数据删除与不可逆操作。

AI 负责提出、实现、审查和验证，不拥有最终产品决策权。

## 2. 开工前必读

每个代理开始任务前必须阅读：

1. `AGENTS.md`
2. `docs/product/PRODUCT_MASTER_PLAN.md`
3. `docs/product/MVP_SCOPE.md`
4. `docs/product/NON_GOALS.md`
5. `docs/management/CHANGE_HANDOFF_LOG.md`
6. 当前任务对应的 GitHub Issue
7. 被修改模块的相关文档和测试

如果文档之间冲突，优先级为：

产品负责人明确指令 > PRODUCT_MASTER_PLAN > MVP_SCOPE > 架构决策记录 > 模块文档 > Issue > 代码注释。

发现冲突时不得自行猜测，应在 Issue 或 PR 中记录冲突并采用最小风险实现。

## 3. 无 Issue 不开工

每项任务必须对应一个 GitHub Issue。Issue 至少包含：

- 背景与用户价值。
- 明确目标。
- 包含范围。
- 不包含范围。
- 涉及模块。
- 验收标准。
- 测试要求。
- 已知风险。

禁止依据一句模糊聊天指令同时实现多个系统。

## 4. 单一主要执行者

同一任务、同一模块、同一时间只能有一个主要执行代理。

推荐流程：

- ChatGPT：产品设计、任务拆解、视觉需求、产品验收。
- Claude：架构方案与高级代码审查。
- Codex：仓库级代码实现、测试、修复和局部重构。
- OpenClaw：任务调度、构建、自动测试、日志整理和状态报告。
- 罗松涛：范围决定、试玩、最终验收和合并批准。

审查者不得在未说明的情况下重写执行者全部方案；执行者不得擅自扩大需求。

## 5. Git 工作流

禁止 AI 直接强制推送或重写 `main` 历史。

推荐分支命名：

- `feature/issue-<id>-<name>`
- `fix/issue-<id>-<name>`
- `docs/issue-<id>-<name>`
- `refactor/issue-<id>-<name>`
- `art/issue-<id>-<name>`

一个 Pull Request 原则上只解决一个 Issue。确需关联多个 Issue 时，必须说明耦合原因。

提交信息使用：

- `feat:` 新功能
- `fix:` 缺陷修复
- `docs:` 文档
- `test:` 测试
- `refactor:` 不改变行为的重构
- `chore:` 工具、配置和维护
- `art:` 美术资产

## 6. 修改边界

AI 可以自主执行：

- 阅读和分析仓库。
- 创建任务分支。
- 在分支内实现已批准 Issue。
- 编写测试和工具。
- 更新与修改同步的文档。
- 创建草稿 PR。
- 提出新 Issue 和风险报告。

必须经产品负责人批准：

- 合并到 `main`。
- 删除大量文件。
- 修改核心产品规则。
- 更换引擎或核心依赖。
- 修改数据库、存档或网络协议。
- 引入收费服务。
- 添加商业化功能。
- 发布公开构建。

永远禁止：

- 将密钥、Token、密码和真实用户数据提交到仓库。
- 绕过测试强行宣称完成。
- 使用未授权的原游戏代码、美术、音乐、角色、Logo 或文本。
- 清空数据库或真实玩家数据。
- 未经批准进行付费、发布和生产环境操作。
- 伪造构建结果、测试结果、截图或日志。

## 7. 代码原则

- 优先可读、可测、可替换，不为炫技引入复杂架构。
- 首个垂直切片阶段采用模块化单体思路，不提前拆分大量微服务。
- 游戏规则使用数据驱动配置，避免散落在 UI 和表现代码中。
- 核心逻辑不得依赖场景对象查找和隐式全局状态。
- 存档、资源、战斗结果和合同结算必须有明确权威来源。
- 所有随机行为必须支持注入随机种子，以便复现测试。
- 不允许在无迁移方案时破坏已有存档。
- 不允许以临时代码为理由跳过输入校验、错误处理和日志。

## 8. 文档同步

修改以下内容时必须同步文档：

- 数据字段和配置结构。
- 公共接口。
- 存档与迁移。
- 游戏规则和数值公式。
- 模块边界。
- 开发、构建和发布流程。

代码与文档不一致时，任务不得标记完成。

## 9. 测试与完成标准

代理不得用“应该可用”“基本完成”“理论没问题”代替证据。

PR 至少说明：

- 完成了什么。
- 没有完成什么。
- 修改了哪些模块。
- 自动测试结果。
- 构建结果。
- 手工验证步骤。
- 已知风险和回滚方式。

具体完成标准见 `docs/management/DEFINITION_OF_DONE.md`。

## 10. 公共交接记录

每个完成或中断的任务必须在 `docs/management/CHANGE_HANDOFF_LOG.md` 增加记录，包含：

- 日期、代理、Issue 和 PR。
- 完成内容。
- 修改文件和模块。
- 未完成内容。
- 关键决策。
- 已知风险。
- 下一位执行者必须知道的事项。

若记录冲突，按提交时间和关联 Issue 判断，并在新的记录中明确纠正，不得静默覆盖。

## 11. 产品边界

所有代理必须遵守产品宪法：

- 玩家从普通人开始。
- 普通伙伴可成长为亲卫、官员和将军。
- 领主、封臣、佣兵和职业武将都必须可形成完整玩法。
- 不强迫所有玩家经营城市。
- 城市布局必须影响战争。
- 深度来自选择而非重复操作。
- 付费不得购买爵位、职位、城市或永久战力碾压。
- 赛季人物故事长期保留，战力与领土不永久滚雪球。

任何与以上原则冲突的实现应停止并提交产品变更提案。

## 12. 范围控制

首个垂直切片前禁止擅自加入：

- 玩家建国与完整国家政治。
- 大规模实时 PVP。
- 万人同图。
- 海战。
- 婚姻家族系统。
- 第三人称骑砍式动作战斗。
- 复杂金融与完整自由市场。
- 跨服战争。
- 真实支付。
- 数百名完整人生模拟 NPC。

优秀的延期功能也仍然是范围膨胀。先证明核心闭环，再扩张世界。
